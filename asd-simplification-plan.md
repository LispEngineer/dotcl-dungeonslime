## Goal Description
The objective of this change is to simplify and refactor [dungeon-slime.asd](file:///home/dfields/src/cl/dotcl-dungeonslime/dungeon-slime.asd) by directly including the subsidiary `cspackages/csharp-assembly-packages.asd` file as a component of the main system, rather than using the complex read-time reader macro splice (`#.(append ... *cspackages-components* ...)`).

In DotCL, compiling a subsidiary `.asd` file as a `:depends-on` dependency fails because dependencies are compiled in an early `_DotCLResolveDeps` MSBuild phase, before the game's referenced .NET assemblies (e.g. `MonoGame.Framework`) are loaded. However, by defining a custom ASDF component class `asd-module` that dynamically reads the subsidiary `.asd` components, we can include `cspackages` as a module component of the main system. This ensures it is compiled in the `_DotCLCompileRoot` phase (where all assemblies are loaded) while keeping the `.asd` definition clean, declarative, and maintainable.

(defmethod asdf::compute-component-children ((m asd-module) components serial-p)
  (declare (ignore components serial-p))
  (let* ((dir (asdf:component-pathname m))
         (asd-path (uiop:subpathname dir (asd-module-file m))))
    (if (uiop:file-exists-p asd-path)
        (let* ((*read-eval* nil)
               (defsystem-form (with-open-file (s asd-path) (read s)))
               (components-list (getf (cddr defsystem-form) :components)))
          (loop
            :with previous-components = nil
            :for c-form :in components-list
            :for c = (asdf::parse-component-form m c-form
                                                  :previous-serial-components previous-components)
            :when c
              :do (let ((name (asdf:component-name c)))
                    (push name previous-components))
              :and :collect c))
        nil)))

          :else
