# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Common Lisp implementation of the MonoGame "Dungeon Slime" 2D tutorial game,
running on [DotCL](https://github.com/dotcl/dotcl) (a Common Lisp environment
hosted on the .NET CLR) plus MonoGame for rendering/input/audio. The code is
primarily Common Lisp (`.lisp`), with a small amount of C# (`.cs`) used only to
smooth over gaps in DotCL/.NET interop. See `README.md` for the full project
narrative and `FILES.md` for a description of every source file.

## Commands

Always prefer the `Makefile` targets below over ad hoc commands.

* `make build` — builds everything (C# and Lisp) in one `dotnet build` step.
  Does not run tests or the game.
* `make test` — runs the built executable with `--test` (does **not** build
  first; use `make build test`).
* `make run` — runs the built executable's GUI game (does **not** build
  first; use `make build run`).
* `make repl` — loads the whole system into a `dotcl` REPL and drops into the
  `:dungeon-slime` package (`*mg-game*` is bound to the game instance).
* `make check-parens` — checks parenthesis balance across all `.lisp`/`.asd`
  files via `check_parens.py`. **Run this immediately after editing any Lisp
  file**, and especially before/after diagnosing any DotCL build failure (see
  Gotchas below).
* `make cspackages` — regenerates `cspackages/` (the generated C# interop
  wrapper packages) using the external `dotcl-packagegen` tool. Requires a
  prior `make build` (bootstraps off the built `MonoGame.Framework.dll`).
  Only needed when adding/using a new .NET class; the generated wrappers are
  committed, so this isn't part of the normal edit/test loop.
* `make mgcb` — launches the MonoGame Content Builder editor for
  `Content.mgcb`.
* `make clean` — `dotnet clean`; does not remove the vendored `cspackages/`.

There is no separate unit test framework — tests are plain Lisp functions run
via `--test`, aggregated in `test-harness.lisp`, printed to stderr. To run
just the tests from a live REPL: `(dungeon-slime-tests::run-all-tests)`.

Font note: MGCB only watches `.spritefont` files, not the raw `.ttf`. If you
edit `Content/fonts/04B_30.ttf`, `touch Content/fonts/04B_30.spritefont`
first or the font won't rebuild.

## Architecture

### Build pipeline: two separate DotCL compile phases

DotCL/MSBuild compiles a project's Lisp in **two distinct phases**, and this
drives several odd-looking patterns in `dungeon-slime.asd`:

1. `_DotCLResolveDeps` — compiles the `:depends-on` system graph
   (`dotnet-class`, `dotcl-thread`, `dotcl-repl`, `anaphora`) standalone,
   *before* this project's own .NET assembly references (e.g.
   `MonoGame.Framework.dll`) are loaded.
2. `_DotCLCompileRoot` — compiles the root system's own `:components` (the
   actual files listed directly in `dungeon-slime`'s component list), *after*
   the project's assemblies are loaded.

Anything that calls `dotnet:resolve-type` at load time (all the generated
`cspackages/*.lisp` wrappers do) must compile in phase 2, not phase 1. That's
why `cspackages/` is never a `:depends-on` system dependency — instead,
`dungeon-slime.asd` `read`s `cspackages/csharp-assembly-packages.asd` (the
generator's own self-contained system definition) at `eval-when` time and
splices its `:components` form directly into `dungeon-slime`'s own
`:components`, prefixed with `"cspackages/"` and forced to depend on
`type-aliases` (the file that actually loads `MonoGame.Framework.dll`). Full
history and the exact failure modes this avoids are in
[doc/implementation-notes.md](doc/implementation-notes.md) under "Wiring
`dungeon-slime.asd` to the Generator's Self-Contained `.asd`".

Consequence: any new file that uses generated `cspackages/` wrapper packages
must list `,@(mapcar (lambda (comp) (second comp)) *cspackages-components*)`
in its `:depends-on` in `dungeon-slime.asd` (see `mg-classes`, `input-manager`,
`audio-controller`, etc. for the pattern).

### Package pre-declaration

`packages.lisp` pre-declares ~40 empty `(defpackage :microsoft-xna-framework-*)`
stubs before the real `cspackages/packages.lisp` (with full exports) loads.
This isn't leftover cruft — it breaks a genuine load-order cycle: this
project's `packages.lisp` needs those package objects to exist so its
`:local-nicknames` (e.g. `(:v2 :microsoft-xna-framework-vector2)`) resolve at
compile time, but the real exports can only load later (phase 2, after
`type-aliases.lisp`). `defpackage` can safely reopen/extend an existing
package, so this works. Do not delete these stubs when adding new cspackages
consumers — add a new stub instead if a genuinely new generated package name
is introduced.

### Generated C# wrappers (`cspackages/`)

All `cspackages/*.lisp` files are generated by the external
[`dotcl-packagegen`](https://github.com/LispEngineer/dotcl-package-generator)
tool (invoked via `make cspackages`) and vendored/committed. They provide
Lispy wrapper functions/constants for MonoGame and System .NET classes (e.g.
`v2:new`, `rect:new`, `game-time:total-game-time`, `sprite-batch:begin`). Game
code should use these wrappers instead of raw `dotnet:invoke`/`dotnet:static`/
`dotnet:new` calls wherever a wrapper exists — this refactor has already been
done project-wide (see "Refactoring Raw .NET Interop Calls" in
`doc/implementation-notes.md`). When extending the game to touch a new .NET
class not yet wrapped, add it to the `--class` list in the `Makefile`'s
`cspackages` target and re-run `make cspackages`, rather than reaching for raw
interop.

### Scene/game architecture

* `core` (`mg-core.lisp`) is the base CLOS class bridging into
  `MonoGameCLOSProxy` (a C# proxy class that forwards MonoGame's `Game`
  callbacks into CLOS methods). It owns the `input-manager` and
  `audio-controller`, background REPL lifecycle, and window setup.
* `scene` (`scene.lisp`) is the abstract base for game scenes, with a
  lifecycle protocol (`initialize`, `load-content`, `update`, `draw`,
  `dispose`) and a scene-private `ContentManager`. `core` holds
  `active-scene`/`next-scene` slots and stages transitions between frames to
  avoid mid-frame scene swaps.
* `title-scene.lisp` and `gameplay-scene.lisp` are the two concrete scenes.
  `game-1` (`game-1.lisp`) is the `core` subclass that bootstraps into
  `title-scene` on startup.
* `main.lisp` instantiates both the CLOS `core`-subclass instance and its C#
  proxy, wiring them together, storing them in `*game*`/`*cs-game*`.

### C# interop conventions

* `csharp.lisp` provides reader macros for terse interop: `#!MethodName obj`
  (instance call), `#!!Class.Method` (static call).
* `clr-generic.lisp` implements `defc#generic`/`defc#method`, a C#-type-aware
  CLOS-like generic dispatch system with topological specificity sorting over
  class/interface inheritance (may be superseded by DotCL 0.1.17's native C#
  type dispatch and `dotnet:class-for-type` in ordinary CLOS methods — check before extending this).
* `monoutils.lisp` / `MonoUtils.cs` add small custom interop helpers DotCL
  lacks natively (e.g. `monoutils:invoke-generic` for generic method calls
  with Lisp type-alias resolution via `dotnet::*type-aliases*`).
* Filesystem paths passed to C# constructors must go through `qualify-path`
  (`utils.lisp`) and `uiop:native-namestring` — raw relative strings or
  `pathname` objects fail or break under different working directories/users.
* C# `float` (single-precision) properties must be set with values explicitly
  `(coerce value 'single-float)`-ed; DotCL marshals Lisp floats to `double` by
  default, and reflection won't narrow `double` to `float` for a setter.
* Host subsystem init (notably audio) is wrapped in `handler-case` so the game
  degrades to silent/no-op mode rather than crashing on restricted
  environments (see "Multi-User Portability" in `doc/implementation-notes.md`).

## Gotchas

* **Mismatched parentheses produce misleading errors.** DotCL concatenates
  all Lisp files in the system into one `.concat.lisp` before compiling, so a
  paren imbalance in one file often surfaces as an unrelated error deep in
  the build (e.g. an MSB3073 exit code 134). Whenever a Lisp build fails
  unexpectedly, run `make check-parens` on all touched files before digging
  further.
* **MSBuild incremental caching hides compile-time output.** If `.fasl`
  outputs are newer than all `.lisp`/`.asd` inputs, `DotclCompileRoot` is
  skipped entirely — `eval-when (:compile-toplevel)` diagnostics won't print.
  Force a real recompile with a trivial edit or `make clean` when debugging
  compile-time macros.
* Don't hardcode the build output path (`bin/Debug/net10.0/<rid>/`) — it
  varies by OS/config. Use `dotnet build DungeonSlime.csproj
  -getProperty:OutputPath` (as the `Makefile` does) or, in Lisp, resolve
  paths dynamically (see "Resolving Build Output Paths Programmatically" in
  `doc/implementation-notes.md`).
* The game can only be `Run()` once per process — a second run segfaults
  (known upstream MonoGame issue). This mainly matters when driving the game
  from a REPL session.

## Conventions from AGENTS.md / GEMINI.md

These files (identical content, targeting other AI tools) define durable
project conventions that also apply here:

* Never use `and`/`or` as control-flow in Common Lisp; use `when`/`if`
  instead.
* C# braces: opening brace stays on the same line as what precedes it
  (including `else`), separated by exactly one space; every `if`/`else`
  block is braced.
* Markdown: no gratuitous double spaces; wrap prose at 80-100 columns; when
  backtick-quoting text that itself contains one backtick, use double
  backticks around it (e.g. ``List`1``).
* Don't use personal pronouns ("we"/"I") in code comments or docs — use
  passive voice or refer to the thing itself.
* Any command run regularly belongs in the `Makefile` as a target.
* Use `scratch/` for throwaway exploration files (git-ignored). Clean up any
  stray `run_out*`/`repl_out*` log files from the repo root before ending a
  session.
* All new code needs tests, using plain Lisp (no test framework in use),
  printed to stderr, run via `--test` or on every startup.
